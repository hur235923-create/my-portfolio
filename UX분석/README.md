# UX 분석 · 개선 문서

포트폴리오 작업물(`../works/`)의 **UX 케이스 스터디 → 개선 설계 → 리디자인 반영** 기록을
작업물별로 모듈화해 보관합니다. 장기적으로 작업물이 늘어나도 같은 구조로 관리합니다.

## 구조

```
UX분석/
├─ README.md            ← 이 파일 (전체 인덱스)
├─ <작업물>/
│  ├─ 1_분석.md          ← UX 케이스 스터디 (10단계)
│  ├─ 2_개선안.md        ← 개선 설계안 (결함 보강)
│  └─ 3_반영.md          ← 리디자인 반영 결과 + 검증
└─ _template/           ← 새 작업물 추가용 빈 양식
   ├─ 1_분석.md
   ├─ 2_개선안.md
   └─ 3_반영.md
```

각 작업물 폴더는 **분석 → 개선안 → 반영**의 3단계 생애주기를 한곳에 모읍니다.
숫자 접두사(1/2/3)로 작업 순서가 드러납니다.

## 문서 3단계

세 문서는 **① 분석 → ② 개선안 → ③ 반영**의 한 줄기로 이어지며, 상호 링크로 연결된다.
브랜드 문제(P1~P5)가 설계·구현까지 추적된다.

1. **① 분석 (1_분석.md)** — 실제 브랜드 사이트를 실측해 UX 문제(P1~P5)를 도출.
   서비스 이해 → 사용자 목표 → 기존 UX 분석(실사이트 실측) → 문제 발견 → 원인 → 문제 정의.
2. **② 개선안 (2_개선안.md)** — 그 문제를 푸는 리디자인 설계(목표·전략, 각 항목에 `→P#`).
   끝에 **추가 결함 보강**(완성 후 발견한 구현 레벨 결함, `문제→근거(파일:라인)→개선→기대→범위`).
3. **③ 반영 (3_반영.md)** — 구현이 P1~P5를 해결한 매핑 + 기대 효과.
   끝에 **추가 결함 보강 반영**(`발견한 문제 / 개선 목표 / 디자인 해결 / 기대 효과 / 검증`).

> '기존' 진단은 2026-07-05 실제 브랜드 사이트(hasselblad.com · jw-pharma.co.kr ·
> yamaha.com · maruni.com/en) 실측 기준. 실측 못 한 부분은 추정하지 않고 '범위 밖'으로 표기.

## 새 작업물 추가 방법

1. `_template/` 폴더를 새 작업물 이름으로 복사한다. 예:
   ```
   cp -r _template <새-작업물명>
   ```
2. `1_분석.md`부터 순서대로 채운다.
3. 이 README의 **작업물 인덱스** 표에 한 줄을 추가한다.

## 참고

- 문서 본문의 `파일:라인` 인용은 작성 시점 기준이므로, 코드 수정 후에는 현재 코드와 대조 후 갱신한다.
- 문서 상단 메타의 옛 파일명 참조(예: `04_...txt`)는 구 평면 구조의 흔적이며 내용은 유효하다.
